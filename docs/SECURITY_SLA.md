# 安全性弱點 SLA

> 最後更新：2026-02-18

## 偵測

弱點透過兩個管道被發現：

- **CI**：`backend-security-audit` 工作項目在每次 PR 和 push 時執行 `composer audit`。
- **排程**：Dependabot 每週（週一 09:00 Asia/Taipei）自動檢查並開啟 PR。

Dependabot 發現的安全性公告會立即觸發 PR，不受每週排程限制。

## 嚴重程度分類

嚴重程度依據 [CVSS v3 基礎分數](https://www.first.org/cvss/v3.0/specification-document) 分類：

| 嚴重程度 | CVSS 分數 | 說明 |
|----------|-----------|------|
| 嚴重 | 9.0 – 10.0 | 遠端程式碼執行、身份驗證繞過、資料遺失 |
| 高 | 7.0 – 8.9 | 權限提升、大量資料外洩 |
| 中 | 4.0 – 6.9 | 影響有限，需額外條件配合 |
| 低 | 0.1 – 3.9 | 影響極小，縱深防禦層級 |

若廠商未提供 CVSS 分數，請使用上方最接近的描述進行對應。

## 回應 SLA

| 嚴重程度 | 分類期限 | 修復期限 | 正式環境部署 |
|----------|----------|----------|--------------|
| 嚴重 | 4 小時 | 24 小時 | 修復後立即部署 |
| 高 | 24 小時 | 48 小時 | 修復後 24 小時內部署 |
| 中 | 3 天 | 7 天 | 下次預定發版 |
| 低 | 7 天 | 30 天 | 下次預定發版 |

「分類期限」= 確認嚴重程度、指定負責人、決定修復路徑。
「修復期限」= PR 合併且 CI 通過。

## 修復流程

1. **偵測**：`composer audit` 或 Dependabot PR 識別出弱點。
2. **分類**：確認嚴重程度與受影響版本。確認專案是否實際使用了含弱點的程式碼路徑。
3. **修復**：透過 `composer update <package>` 更新至已修補的版本。提交更新後的 `composer.lock`。
4. **驗證**：CI 必須通過（`backend-ci` + `backend-security-audit` + `backend-coverage`）。
5. **部署**：依照上方嚴重程度 SLA 執行正式環境上線。
6. **結案**：將 Dependabot PR 或 Issue 標記為已解決。在相關 DEVLOG 中記錄修復內容。

## 例外情況

若無可用的修補版本：

- 實施暫行方案，並在 `docs/security-exceptions.md` 中新增記錄。
- 待修補版本釋出後，在原始 SLA 時間內重新評估。
- 無可用修補版本的「嚴重」與「高」等級弱點，必須立即升級處理。

## 範圍

本 SLA 適用於 `ledger-of-decisions-api`（PHP/Composer 套件相依性）。
前端與 E2E 的套件安全性透過 Dependabot 追蹤，但遵循團隊層級的分類流程。
